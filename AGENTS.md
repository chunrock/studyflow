# AGENTS.md — 바이브코딩 표준 에이전트 규칙

이 문서는 웹 자료를 생성·미리보기·갱신하는 모든 프로젝트에서 코딩 에이전트(Claude Code, Codex 등)가 지켜야 할 **공통 기준**입니다. 새 프로젝트를 시작할 때 이 파일을 프로젝트 루트에 복사하고, 프로젝트별 항목(서버 이름·포트·원본 폴더)만 채워서 사용합니다.

---

## 1. 기본 원칙: 카파시룰 (Karpathy 바이브코딩 원칙)

모든 작업은 아래 원칙을 **기본값**으로 따른다. 이 원칙은 설치된 플러그인 `andrej-karpathy-skills`의 `karpathy-guidelines` 스킬(Andrej Karpathy가 정리한 LLM 코딩 실수 방지 가이드라인, MIT)을 기준으로 한다. Claude Code에서 작업할 때는 코드 작성·리뷰·리팩토링 시 이 스킬을 활성화해 사용한다.

### 1-1. 코딩 전에 생각한다 (Think Before Coding)

**가정하지 말고, 혼란을 숨기지 말고, 트레이드오프를 드러낸다.**

- 구현 전에 자신의 가정을 명시적으로 밝힌다. 불확실하면 질문한다.
- 해석이 여러 가지면 조용히 하나를 고르지 말고 선택지를 제시한다.
- 더 단순한 방법이 있으면 말한다. 필요하면 사용자 요청에도 반론을 제기한다.
- 불명확한 것이 있으면 멈추고, 무엇이 혼란스러운지 짚고, 질문한다.

### 1-2. 단순함 우선 (Simplicity First)

**문제를 푸는 최소한의 코드만. 추측성 코드는 금지.**

- 요청받지 않은 기능을 추가하지 않는다.
- 한 번만 쓰는 코드에 추상화를 만들지 않는다.
- 요청받지 않은 "유연성"·"설정 가능성"을 넣지 않는다.
- 일어날 수 없는 시나리오에 대한 에러 처리를 넣지 않는다.
- 200줄을 썼는데 50줄로 가능하면 다시 쓴다.
- 자문한다: "시니어 엔지니어가 보면 과하게 복잡하다고 할까?" 그렇다면 단순화한다.
- 이 기준에서는: 프레임워크·빌드 도구 없이 정적 HTML/CSS/JS로 시작하고, 필요가 증명된 뒤에만 도구를 추가한다.

### 1-3. 외과수술식 변경 (Surgical Changes)

**꼭 필요한 부분만 건드리고, 내가 만든 흔적만 치운다.**

- 변경 대상 주변의 코드·주석·포맷을 "개선"하지 않는다.
- 망가지지 않은 것을 리팩토링하지 않는다.
- 내 취향과 달라도 기존 스타일을 따른다.
- 무관한 죽은 코드를 발견하면 삭제하지 말고 언급만 한다.
- 내 변경으로 안 쓰이게 된 import/변수/함수는 제거하되, 원래 있던 죽은 코드는 요청 없이는 제거하지 않는다.
- 판정 기준: 변경된 모든 줄이 사용자의 요청으로 직접 설명돼야 한다.

### 1-4. 목표 중심 실행 (Goal-Driven Execution)

**성공 기준을 정의하고, 검증될 때까지 반복한다.**

- 작업을 검증 가능한 목표로 바꾼다. 예: "버그 수정" → "버그를 재현하는 테스트를 만들고 통과시킨다".
- 여러 단계 작업은 단계마다 `[단계] → 검증: [확인 방법]` 형태의 짧은 계획을 먼저 밝힌다.
- 이 기준에서는: 웹 자료 변경의 성공 기준은 **미리보기 화면**이다. 변경 직후 미리보기(스냅샷·콘솔 에러)로 검증한 뒤 다음 단계로 넘어가고, 검증 없이 변경을 쌓지 않는다.
- 콘솔 에러·서버 로그는 에이전트가 직접 읽고 원인을 찾아 고친 뒤 결과만 보고한다.

> 트레이드오프: 이 원칙들은 속도보다 신중함에 무게를 둔다. 사소한 작업에는 판단에 따라 완화한다.

## 2. 작업 프로세스

모든 작업은 아래 4단계 사이클로 진행한다.

```
① 자료 확인 → ② 웹 자료 생성 → ③ 실시간 미리보기 → ④ 요청 시 갱신 (③으로 복귀)
```

### ① 자료 확인
- 작업 시작 전 프로젝트 폴더의 **현재 생성 자료**(기존 HTML/CSS/JS, 참고자료, 문서)를 먼저 파악한다.
- 기존 자료가 있으면 그 구조·스타일·네이밍을 따르고, 처음부터 새로 만들지 않는다.

### ② 웹 자료 생성
- 프론트엔드 자료(HTML, CSS, JS, 폰트, 정적 파일)는 항상 **원본 폴더 하나**(`web/` 권장)에만 생성·수정한다.
- 빌드 사본, 임시 미리보기 복사본(`/tmp/...`)은 원본이 아니다. 항상 원본을 수정한 뒤 사본을 동기화한다.
- 새 프로젝트를 시작할 때는 원본 폴더와 미리보기 서버 정보를 먼저 정한다. 정하지 않은 상태에서 여러 위치에 파일을 만들지 않는다.

### ②-0. 원본 폴더와 실행기 폴더 분리
- 웹 자료의 수정 원본은 항상 `web/` 같은 원본 폴더에 둔다.
- `exe/`, `app/`, `server/`, `dist/`, `webroot/`처럼 실행·배포·임베드에 쓰는 폴더는 관제 또는 산출물 폴더로 취급한다.
- 실행기 폴더에서는 원본 자료를 직접 수정하지 않는다. 필요한 경우 `sync`, `generate`, `build`, `preview` 명령으로 원본을 복사·임베드·검증한다.
- 원본에서 실행기 산출물로 복사되는 구조라면 복사 방향을 문서화한다. 예: `web/ → exe/webroot/ → 실행 파일 embed`.
- 실행기 폴더에 `dev.sh`, `Makefile`, `taskfile` 같은 관제 스크립트를 둘 수 있다. 이 스크립트는 동기화, 미리보기, 검증, 빌드를 한 곳에서 실행하기 위한 진입점이어야 한다.
- 관제 스크립트는 프로젝트의 등록 포트 1개만 사용해야 하며, 포트가 사용 중이면 다음 빈 포트를 자동 선택하지 말고 기존 프로세스를 재사용·정리하거나 중단한다.

### ②-0-1. web → webroot → embed 동기화 시 누락 방지 (Go 프로그램)

`web/`을 Go 실행 파일에 `//go:embed`로 내장하는 프로젝트는 **검진팀 대시보드 방식(전체 복사형)을 표준으로 한다.** 파일명을 하나씩 지정하는 allowlist 방식은 새 파일·폴더를 목록에 추가하지 않으면 에러 없이 조용히 빠지므로 쓰지 않는다.

```go
//go:generate sh -c "rm -rf webroot && cp -R ../web webroot && find webroot -name .DS_Store -delete"
//go:embed all:webroot
```

- `web/` 전체를 그대로 복사하므로 새 파일·폴더를 추가해도 누락되지 않는다. `all:` 접두사는 필수(없으면 점·밑줄로 시작하는 파일이 빠진다).
- `go build`를 직접 호출하지 않고 항상 `build.sh`(`go generate ./...` 선행 포함)를 거친다.
- 빌드 후 5-1의 "실행기 검증 방법"으로 실제 실행 파일에 반영됐는지 한 번 확인한다.

### ②-1. 자료 모듈화
- 화면에 들어가는 원본 데이터·문구·설정은 가능한 한 HTML 구조와 분리한다.
- 반복되는 항목(일정, 표 행, 카드, FAQ, 탭 콘텐츠 등)은 `data/`, `content/`, `modules/` 중 프로젝트의 기존 폴더 규칙에 맞춰 별도 파일로 관리한다.
- 기존 프로젝트에 데이터 모듈 규칙이 있으면 그 규칙을 따른다. 없으면 정적 자료는 `web/data/`, 화면 조립 JS는 `web/scripts/`에 두는 것을 기본값으로 한다.
- 같은 내용을 HTML, JS, JSON에 중복으로 적지 않는다. 한 곳의 원본을 읽어 화면을 렌더링한다.
- 모듈을 나눌 때는 실제로 재사용되거나 갱신 단위가 분리되는 경우에만 나눈다. 한 번만 쓰는 작은 문구까지 과하게 분리하지 않는다.
- 첨부 자료를 반영할 때는 원문 의미를 보존하고, 요약·재배열·표준화한 내용은 데이터 파일에서 구조가 드러나게 둔다.

### ③ 실시간 미리보기
- 아래 **3. 서버 규칙**에 따라 **등록된 서버 1개**로만 미리보기한다.
- 정적 파일이므로 서버 재시작 없이 브라우저 새로고침으로 변경을 확인한다.
- UI 변경 후에는 스냅샷/스크린샷과 콘솔 에러를 확인한 뒤 완료를 보고한다.

### ④ 요청 시 갱신
- 사용자의 수정 요청을 받으면 원본 폴더의 자료를 갱신하고, 실행 중인 서버를 그대로 재사용해 ③으로 돌아간다.
- 서버를 새로 띄우거나 포트를 바꾸지 않는다.

## 3. 개발/미리보기 서버: 프로젝트당 반드시 1개만 실행

자원 낭비를 막기 위해 웹 미리보기 서버는 **프로젝트당 동시에 1개만** 띄운다.

- 서버가 필요하면 새로 만들지 말고, **이미 실행 중인 서버가 있는지 먼저 확인**하고 재사용한다.
- 임의로 `python3 -m http.server <port>` 같은 즉석 명령을 띄우지 않는다. 반드시 아래 "등록된 서버"만 사용한다.
- Claude Code에서는 `.claude/launch.json`에 서버를 등록하고 프리뷰 도구(preview_start)로 실행한다. 프리뷰 도구는 동일 서버가 실행 중이면 자동으로 재사용한다.

### 등록된 서버 (프로젝트별로 채울 것)

- 이름: `studyflow`
- 포트: `5187`
- 정의 위치: `.claude/launch.json` (2026-07-22 결정: `web/` 정적 미리보기 전용, `tools/static-server.js`. Electron 실행은 `npm run dev`로 별도 유지)
- Codex 등 다른 도구에서 서버를 띄워야 할 때도 **같은 포트 하나만** 사용하고, 다른 포트로 같은 프로젝트를 중복 서빙하지 않는다.

### Codex에서 작업할 때

- `.claude/launch.json`이 있으면 그 안의 이름·포트·명령을 기준으로 한다.
- `.claude/launch.json`이 아직 없으면 새 임시 포트를 고르기 전에 사용자에게 프로젝트의 고정 포트를 확인하거나, 프로젝트 문서에 포트 결정을 남긴다.
- 로컬 복사본으로 서빙하는 경우에도 서버는 1개만 유지하고, 원본 수정 후 `rsync`로 복사본을 갱신한다.

### 새 서버를 실행하기 전 확인 절차

```sh
# 1) 이 프로젝트를 이미 서빙 중인 프로세스가 있는지 확인
lsof -iTCP -sTCP:LISTEN -P 2>/dev/null
ps aux | grep -E "http.server|node .*serve|preview" | grep -v grep

# 2) 의심되는 PID의 작업 디렉토리(cwd)를 확인해 이 프로젝트를 가리키는지 검증
lsof -p <PID> 2>/dev/null | grep cwd
```

- cwd가 이 프로젝트를 가리키는 중복/유령 프로세스가 있으면 먼저 종료(kill)한 뒤 등록된 서버 1개만 남긴다.
- 작업 세션이 끝나도 사용자가 요청하기 전에는 서버를 임의로 여러 개 재기동하지 않는다.
- 등록된 포트가 이미 사용 중이라는 오류가 나면, `lsof -iTCP:<포트> -sTCP:LISTEN -P`로 점유 PID를 찾고 `ps -p <PID> -o pid,lstart,command`로 명령을 확인한다. Go/Rust/Java 등 컴파일 언어 실행기는 `go run`/`cargo run` 등으로 뜬 임시 빌드 바이너리(예: 경로에 `go-build...` 포함, 또는 프로젝트 바이너리명과 동일한 프로세스명)로 포트를 물고 있는 경우가 많다. cwd 확인 없이 바이너리명·경로만으로도 이 프로젝트 소유임을 알 수 있으면 바로 종료한다.

### 참고: SynologyDrive 마운트 이슈

- SynologyDrive 마운트 경로를 직접 서빙하면 환경에 따라 응답이 없거나(hang) stale하게 서빙될 수 있다.
- 이 경우 `/tmp/<프로젝트명>_preview` 같은 로컬 복사본을 만들어 서빙하되, 이 역시 **등록된 서버 1개**로만 운영하고 원본 마운트 경로를 별도로 또 서빙하지 않는다.
- 원본 수정 후에는 반드시 로컬 복사본을 최신화한다:

```sh
rsync -a --delete web/ /tmp/<프로젝트명>_preview/
```

## 4. 테마·디자인 기준

- 웹 자료의 색상·폰트·레이아웃·컴포넌트 규칙은 [THEME.md](THEME.md)를 따른다.
- 표준 원본은 **검진팀 대시보드**(`검진팀 대시보드/dashboard/web/styles/`)의 테마이며, 색상·치수는 `theme.css`의 `:root` 변수로만 정의하고 하드코딩하지 않는다.
- 기존 프로젝트(예: 종합검진센터 스케쥴관리)가 자체 토큰을 쓰고 있으면 그 프로젝트 안에서는 기존 토큰을 따르고, 표준으로의 통일은 사용자가 요청할 때만 진행한다.

## 5. 갱신 자료 검증

- JS 파일 수정 후에는 `node --check <파일>`로 문법을 검증한다.
- HTML/CSS 변경은 실행 중인 미리보기 서버에서 직접 확인한다(콘솔 에러, 스냅샷/스크린샷).
- 변경 이력은 프로젝트의 `CHANGELOG.md`에 한 줄씩 기록한다.
- 문서만 수정한 경우에는 링크 대상, 코드블록, 파일명 예시가 실제 구조와 맞는지 확인하고 `CHANGELOG.md`에 기록한다.

### 5-1. 정적 미리보기와 실행기 검증 기준
- 일반 UI 수정(HTML/CSS/클라이언트 JS/정적 데이터)은 `web/` 원본 기준 미리보기, 콘솔 에러 확인, JS 문법 검사로 검증한다.
- 저장소, 로그인, 세션, 설정 파일, 서버 API, 파일 업로드·다운로드, 백업·복원, 외부 프로그램 호출처럼 실행기나 서버 기능을 거치는 변경은 실행기 기준으로도 확인한다.
- 실행기 코드, 임베드 설정, 동기화 스크립트, 빌드 스크립트를 수정한 경우에는 정적 미리보기만으로 완료하지 않는다. 동기화 후 실행기 실행 또는 빌드까지 확인한다.
- 배포·전달 직전에는 최종 산출물 기준으로 1회 실행 확인한다.
- 항상 실행기에서 테스트하는 것은 과할 수 있다. 다만 사용자에게 전달되는 환경이 실행기라면 최종 검증 기준은 실행기 산출물이다.

**실행기 검증 방법 (격리된 임시 실행)**: 등록된 미리보기 서버(정적 파일 서버)는 실행기의 서버 기능(로그인, API, 파일 저장 등)을 대신 검증해 주지 않는다. 이런 변경은 아래 절차로 실제 실행기를 임시로 띄워 확인한다.
```sh
# 1) 원본 폴더 전체(웹+실행기)를 스크래치 경로로 복사 — 원본을 건드리지 않음
cp -R web/  <scratch>/project/web/
cp -R exe/  <scratch>/project/exe/
# 2) 스크래치 실행기 폴더에서 빌드(go generate 등으로 web→webroot 동기화 포함)
cd <scratch>/project/exe && go generate ./... && go build -o /tmp/<name>_test .
# 3) 등록된 미리보기 포트와 겹치지 않는 임시 포트로 백그라운드 실행
(/tmp/<name>_test > /tmp/<name>_test.log 2>&1 &)
# 4) curl로 실제 API/인증/파일 흐름을 직접 확인 (필요한 만큼 반복)
curl -s http://127.0.0.1:<임시포트>/api/...
# 5) 확인 후 프로세스 종료 + 스크래치 파일·바이너리·로그 전부 삭제
pkill -f /tmp/<name>_test; rm -rf /tmp/<name>_test* <scratch>/project
```
- 이 방식은 등록된 서버(3장)와 별개의 **완전히 격리된 1회성 프로세스**이므로 "서버는 프로젝트당 1개" 규칙을 위반하지 않는다. 검증이 끝나면 반드시 정리한다.
- 저장 파일(setting 폴더 등)이 실제로 생성·기록되는지, 응답 상태 코드가 기대대로 나오는지까지 확인해야 "검증됨"으로 본다. 코드를 읽고 "논리적으로 맞다"고 판단하는 것만으로는 충분하지 않다.

### 5-2. 빌드·버전 관리 규칙
- 빌드 전에 프로젝트의 단위 테스트가 있으면 반드시 실행하고 통과를 확인한다(JS: `npm test` 등, Go: `go vet ./... && go test ./...`). 실패한 채로 빌드하지 않는다.
- 빌드 산출물은 버전 폴더(예: `application/v<버전>/`)로 구분한다. 사용자가 특정 버전 번호로 빌드해 달라고 요청했는데 그 버전 폴더에 **이미 실제 이전 릴리스 산출물이 존재**하면, 덮어쓰면 그 이력이 사라진다는 점을 먼저 알리고 진행 여부를 확인한다. 사용자가 명시적으로 덮어쓰기를 승인한 경우에만 진행한다.
- 빌드가 끝나면 `CHANGELOG.md`에 해당 버전 섹션(또는 "빌드" 항목)으로 산출물 크기와 SHA-256 해시를 기록한다. 같은 버전 번호로 재빌드한 경우에도 매번 새 해시를 기록해, 같은 버전 태그 아래 실제로 어떤 산출물이 배포됐는지 추적 가능하게 한다.
- 버전 번호를 올리지 않고 재빌드하는 것은(예: "버전 변경 없이 빌드") 사용자가 명시적으로 요청했을 때만 한다. 별도 언급이 없으면 이미 배포된 버전을 다시 빌드하기보다 다음 버전 번호로 새로 발행하는 쪽을 기본으로 제안한다.

### 5-3. 데이터 변경 전 백업 규칙

DB·엑셀·설정 파일처럼 실제 업무 데이터를 담은 파일을 덮어쓰는 작업은, 손상 시 되돌릴 수 없으므로 **쓰기 전에 항상 백업을 먼저 만든다.**

- **에이전트가 직접 실데이터를 건드릴 때**(버그 재현, 업로드·복구 테스트, DB 수정 등): 원본 파일을 건드리기 전에 타임스탬프가 붙은 사본을 만든다.
  ```sh
  cp company.xlsx company.xlsx.bak-$(date +%Y%m%d%H%M%S)
  ```
  가능하면 실데이터 대신 스크래치 사본에서 재현하고(5-1 "실행기 검증 방법" 참고), 실데이터 경로를 직접 대상으로 테스트하지 않는다.
- **저장 로직을 구현·수정할 때**: 업로드·가져오기처럼 외부 입력으로 기존 파일을 덮어쓰는 기능은 **검증을 먼저 통과한 뒤에만 저장**하도록 순서를 짠다. 임시 파일에 먼저 쓰고 검증 후 원본 자리로 교체(rename)하는 방식을 기본으로 하며, 검증 전에 원본 경로에 바로 write하지 않는다.
- 위 두 규칙 중 하나라도 어겨 실제 데이터가 손상된 사고가 발생하면, 복구 여부와 무관하게 사고 경위·복구 방법을 사용자에게 먼저 보고한다.

## 6. 새 프로젝트 적용 체크리스트

새 프로젝트 루트에 이 기준을 복사한 뒤 아래 항목을 먼저 채운다.

- `AGENTS.md`: 프로젝트명, 원본 폴더, 서버 이름, 포트
- `.claude/launch.json`: 등록된 서버 1개
- `web/styles/theme.css`: [THEME.md](THEME.md)의 표준 토큰 또는 기존 프로젝트 토큰
- `web/fonts/`: 이 폴더의 [fonts/](fonts/) 복사본
- `web/data/` 또는 기존 데이터 폴더: 반복 콘텐츠·첨부 자료 기반 구조화 데이터
- 실행기/배포 폴더가 있는 경우: 원본에서 산출물로 가는 동기화 방향과 관제 명령
- `CHANGELOG.md`: 최초 생성 및 이후 변경 기록

## 7. 하네스 에이전트 자료

이 저장소는 StudyFlow 앱 자료와 함께 Claude Code/Codex 하네스 구성을 재사용 가능한 문서로 관리한다. 실제 `~/.claude` 설정을 직접 변경하기 전에는 저장소의 템플릿을 검토하고 필요한 대상만 복사한다.

- 전역 규칙 템플릿: `agents/global/CLAUDE.md`
- 설정 템플릿: `agents/global/settings.template.json`
- 팀 구성: `agents/teams/*/config.json`
- 팀 멤버 문서: `agents/teams/*/agents/*.md`
- 하위 프로젝트 템플릿: `subprojects/*.md`

### 하네스 자료 갱신 원칙

- 되돌리기 어려운 변경은 실행 전에 변경 대상과 백업 방식을 명시한다.
- API 키, 토큰, 인증서, 개인 경로 같은 비밀정보는 문서에 저장하지 않는다.
- 실제 프로젝트 경로는 `<PROJECT_ROOT>` 같은 플레이스홀더로 유지한다.
- 팀 구성 변경 시 관련 하위 프로젝트 문서도 함께 갱신한다.
